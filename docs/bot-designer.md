# Bot Designer 用户手册

> 面向代理商运营人员的 Telegram 机器人可视化配置操作手册。
>
> 通过本工具，你可以在网页端完成机器人菜单按钮、消息模板、Token 的全部配置，**无需写一行代码、无需联系开发**，修改保存后下一条用户消息即生效。

---

## 快速开始（5 步走完）

1. **登录后台**：使用代理商账号登录 energybot 管理后台。
2. **进入配置页**：左侧菜单点击「机器人控制 → 机器人配置」。
3. **配置 Token**：在「基础设置」Tab 粘贴从 [@BotFather](https://t.me/BotFather) 获取的 Telegram Bot Token，点击「保存基础设置」。
4. **设计菜单**：切到「菜单设计」Tab，从左侧组件库拖入按钮到中间画布，点击选中后在右侧属性面板配置。
5. **开启机器人**：回到页面顶部的「运行状态」卡片，点击「开启机器人」按钮，完成。

## 1. 功能概览

「机器人配置」页由 3 个 Tab 组成：

| Tab | 职责 | 保存行为 |
|---|---|---|
| 基础设置 | Bot Token、机器人用户名、状态、备注 | 点击「保存基础设置」按钮提交 |
| 菜单设计 | 可视化设计按钮树（支持嵌套子菜单） | 点击画布顶部「保存菜单」按钮提交 |
| 消息模板 | 编辑 9 种场景的 Bot 回复文案 | 修改后自动保存 |

**页面顶部的「运行状态」卡片**与所有 Tab 共享，随时可查看 Bot 是否在线、最近心跳、启动/停止时间等信息，并提供一键「开启 / 关闭机器人」按钮。

### 1.1 核心能力

- **所见即所得**：菜单设计器右侧提供 Telegram 风格实时预览，消息模板编辑区有变量高亮预览。
- **乐观锁保护**：多人同时编辑同一机器人时，后提交的一方会收到「配置已被他人修改」提示，避免静默覆盖。
- **保存前预检**：菜单保存前会自动做一次校验（深度、套餐 ID 归属等），在真正写库前就能暴露问题。

## 2. 基础设置

> [截图：基础设置 Tab 的表单]

### 2.1 字段说明

| 字段 | 是否必填 | 说明 |
|---|---|---|
| 机器人状态 | 是 | `启用` / `停用`。停用后 Bot 不会响应用户消息，但配置仍保留。 |
| Telegram Bot Token | 首次配置必填 | 从 [@BotFather](https://t.me/BotFather) 的 `/newbot` 流程获取，格式形如 `123456:ABC-DEF...`。**留空表示不修改**，已有 Token 会被保留。右侧标签会显示「已配置 / 未配置」状态。 |
| 机器人用户名 | 可选 | 如 `@your_bot`，用于在消息模板变量 `{botName}` 中自动填充。 |
| 备注 | 可选 | 内部备注，不对外展示。 |

### 2.2 Token 获取步骤

1. 打开 Telegram 搜索 `@BotFather` 并发起对话。
2. 发送 `/newbot`，按提示依次输入「机器人显示名称」和「机器人用户名（必须以 `_bot` 结尾）」。
3. BotFather 返回形如 `123456789:AAEabcde...XYZ` 的 Token，复制后粘贴到「Telegram Bot Token」输入框。
4. 点击「保存基础设置」，页面会自动重新拉取最新的运行状态。

> **重要**：Token 输入框是密码型字段，保存后本地不再回显；如需更换 Token，重新粘贴新值并保存即可，留空则保留旧值。

### 2.3 一键启停

页面顶部「运行状态」卡片有两个按钮：

- **开启机器人**：前置条件是 Token 已配置，否则按钮会置灰。点击后 Bot 进程接管并开始 Telegram 长轮询。
- **关闭机器人**：立即停止轮询，用户发送消息将无响应。期望状态持久化，下次后端重启会按此期望状态恢复。

## 3. 菜单设计

菜单设计器是本工具的核心模块，用于可视化编排用户在 Telegram 里看到的按钮树。

### 3.1 四栏布局

> [截图：菜单设计器四栏布局全貌]

从左到右：

| 栏位 | 作用 |
|---|---|
| **组件库** | 9 种可拖拽的按钮原型卡片。 |
| **画布** | 按钮的编辑区，顶部是面包屑导航（展示当前所在层级），中部是行/按钮网格。 |
| **属性面板** | 选中一个按钮后，在此编辑它的文本、动作、URL、消息、样式等属性。 |
| **Telegram 预览** | 按 Telegram 原生样式实时预览当前菜单的展示效果。 |

画布上方还有一个**工具栏**，提供以下操作：

- **保存菜单**：把当前画布上的菜单树提交到后端。
- **重置**：放弃所有未保存的改动，恢复到最近一次保存的状态。
- **撤销 / 重做**：针对当前编辑会话内的修改按顺序回退或重做。
- **有未保存更改**：右侧会出现提示标签，提醒用户及时保存。

### 3.2 9 种按钮类型

下表权威源来自 `designer/types.ts` 的 `ButtonAction` 枚举与 `action-icons.ts` 的标题映射。

| 动作值 | 中文标题 | 图标 | 用途 | 配置字段 | 使用场景 |
|---|---|---|---|---|---|
| `url` | 网址链接 | link | 点击后跳转到外部网址 | `URL 地址` | 跳转客服 Telegram 主页、官网、帮助中心链接等 |
| `text` | 文本消息 | message | 点击后 Bot 回复一段预设文字 | `消息内容` | 展示使用说明、套餐价格表、免责声明等静态文案 |
| `command` | 命令 | code | 触发 Bot 的 `/xxx` 内置命令 | `命令`（如 `/start`、`/help`） | 让用户通过按钮代替手敲命令，降低门槛 |
| `start` | 开始 | home | 返回欢迎界面（主菜单） | 无 | 在子菜单里放一个「返回主菜单」按钮 |
| `submenu` | 子菜单 | folder | 下钻到下一级菜单 | 无（子菜单内容通过「双击进入」编辑） | 对按钮分组，如「能量购买」下再分「5 万 / 10 万 / 50 万」套餐 |
| `energy_package_group` | 能量套餐组 | thunderbolt | 展示当前代理商下的一组能量套餐 | `套餐列表` + `排序方式` + `文本模板` | 让用户直接从套餐列表里选购 |
| `address_manage` | 地址管理 | environment | 打开地址管理面板 | 无 | 让用户增/删/切换收能量的 TRON 地址 |
| `wallet_query` | 钱包查询 | wallet | 查询用户钱包链上数据 | 无 | 展示 TRX 余额、带宽、能量等信息 |
| `orders` | 我的订单 | ordered-list | 展示用户订单列表 | 无 | 让用户查看自己的历史订单 |

> **配置示例**：
>
> - 客户想查订单历史 → 在主菜单放一个「我的订单」按钮，动作选 `orders`，无需其他配置。
> - 希望在 Bot 里提供在线客服入口 → 放一个「联系客服」按钮，动作选 `url`，URL 填客服 Telegram 主页。
> - 把「能量购买」按套餐规格分组 → 主菜单放一个「能量购买」按钮（`submenu`），双击进入后再拖入若干「能量套餐组」按钮，分别配置不同的套餐 ID 列表。

#### 能量套餐组（`energy_package_group`）详解

这是最强大也最需要谨慎配置的按钮类型。属性面板提供三个字段：

- **套餐列表**：从下拉组件选择当前代理商名下的套餐，支持多选与拖拽排序。**只能选属于当前 agent 的套餐**，否则保存时后端会以 400 错误拒绝。
- **排序方式**：`price_asc`（价格升序，默认） / `price_desc`（价格降序） / `manual`（按套餐列表里手动拖拽的顺序）。
- **文本模板**：每一行套餐按钮显示的文案模板，支持 `{name}`、`{price}` 等占位符，例如 `{name} - {price} TRX`。

### 3.3 操作说明

#### 3.3.1 新增按钮（拖拽）

1. 把鼠标指向组件库中的目标卡片（如「网址链接」）。
2. 按住左键拖动卡片到画布中央。
3. 松开到：
   - **已有行内**：在该行末尾追加按钮（前提是该行按钮数未达到上限）。
   - **「新增一行」落点**：自动新建一行并把按钮放进去（前提是总行数未达到上限）。

> **提示**：也可以直接点击「新增一行」灰色按钮手工开一行空行，再拖按钮进去。

#### 3.3.2 编辑按钮

1. 在画布里**单击**按钮卡片（卡片会高亮表示选中）。
2. 右侧属性面板会自动展开，根据按钮的动作类型显示对应的字段。
3. 修改字段后会**立即生效**到画布和预览（未保存状态下），等待你点「保存菜单」提交。

#### 3.3.3 删除按钮

点击按钮卡片右上角的**垃圾桶图标**，会弹出确认气泡「确定删除该按钮？」，点「删除」即可。

#### 3.3.4 进入 / 退出子菜单

- **进入子菜单**：双击 `submenu` 类型的按钮卡片（卡片右侧有右箭头提示图标）。画布切换到子菜单层级，面包屑增加一项。
- **回到上层**：点击**面包屑**中任意一级，画布立即跳回该层。

#### 3.3.5 调整按钮顺序（当前限制）

> **⚠️ 当前版本限制**：同画布内的按钮排序、跨行移动**尚未支持**——画布里的拖拽事件只识别来自「组件库」的新建动作。调整顺序的可选做法：
>
> - **删除 + 重新拖入**：删掉目标按钮，再从组件库拖新的进来（属性会丢失，需重填）。
> - **等待后续版本**：同画布拖拽排序已在开发计划中。

### 3.4 容量限制

下表上限值权威源来自 `designer/types.ts` 的 `MAX_*` 常量。

| 约束 | 上限 | 说明 |
|---|---|---|
| 菜单嵌套深度 | 3 层 | 根菜单算 1 层，每进一层 `submenu` +1 层。第 3 层里不能再放 `submenu` 类型按钮。 |
| 每个菜单行数 | 8 行 | 单层菜单内最多 8 行。达到上限后「新增一行」按钮会置灰。 |
| 每行按钮数 | 4 个 | 单行最多 4 个按钮。达到上限后，继续往该行拖按钮会被忽略。 |
| 按钮文本长度 | 64 字符 | 超出会触发属性面板的错误提示，无法保存。 |

### 3.5 保存与校验

#### 3.5.1 保存流程

1. 点击「保存菜单」按钮。
2. 前端先做一次本地校验（深度、文本长度、URL 格式等），通过后提交到后端。
3. 后端再次校验并写库：
   - 成功 → 右上角弹出「操作成功」提示，「有未保存更改」提示消失。
   - 失败 → 工具栏右侧显示红色错误提示 banner。

#### 3.5.2 常见保存失败原因

- **菜单嵌套深度不能超过 3 层**：通常是第 3 层里放了 `submenu` 类按钮，把它改成其他类型或移到第 1、2 层。
- **套餐 ID 不属于当前 agent**：多半是此前从别的 agent 拷贝了配置，或套餐在后台被删除。编辑该按钮的「套餐列表」，重新选有效套餐即可。
- **按钮文本超长**：缩短到 64 字符内。
- **URL 格式不合法**：补全 `https://` 前缀或修正拼写。

#### 3.5.3 乐观锁冲突（409）

如果你打开页面后，**另一个同事也在编辑同一个 agent 的配置**并先你一步保存，你再保存时会收到「配置已被他人修改」提示。**这是一道安全阀**，防止你的改动静默覆盖对方。处理方式：

1. 记住当前画布上你还未保存的改动。
2. 点工具栏的「刷新」或整页 F5，拉取最新配置。
3. 基于最新配置重新改一遍，再保存。

## 4. 消息模板

消息模板编辑器用于定制 Bot 在 9 种关键场景下的回复文案。

> [截图：消息模板编辑器左侧 Tab + 中部编辑区 + 右侧预览]

### 4.1 界面布局

- **左侧**：9 个场景的垂直 Tab 栏。Tab 标题右上出现红点表示该场景文案中含有未识别的变量（需要修正）。
- **中上**：可插入变量列表——12 个变量的徽章，**推荐当前场景使用**的变量会高亮。
- **中下**：左右分栏。左侧是文案编辑 textarea；右侧是实时渲染预览。

### 4.2 9 个场景

下表权威源来自 `template-constants.ts` 的 `SCENE_METADATA`，顺序按用户生命周期排列。

| 场景 Tab | 对应字段 | 触发时机 | 默认文案（留空时使用） | 推荐变量 |
|---|---|---|---|---|
| 欢迎/主菜单 | `welcome` | 用户发送 `/start` 或点击主菜单 | `你好，欢迎使用 {botName}，请选择操作：` | `botName` |
| 未识别命令 | `unknownCommand` | 用户发送机器人无法识别的命令 | `命令 {command} 无法识别，请使用菜单操作。` | `command` |
| 套餐不可用 | `packageUnavailable` | 用户点击的套餐已下架或停售 | `套餐 {packageName} 暂不可用，请选择其他套餐。` | `packageName` |
| 地址无效 | `addressInvalid` | 用户输入的 TRON 地址格式或校验不通过 | `地址 {address} 格式不正确，请输入合法的 TRON 地址。` | `address` |
| 订单创建成功 | `orderCreated` | 用户提交订单后的确认消息 | 见源文件（多行订单明细） | `orderNo`、`packageName`、`amount`、`energy`、`address`、`payAddress` |
| 支付等待中 | `payPending` | 订单已提交，等待链上确认 | `订单 {orderNo} 正在确认中，请耐心等待。` | `orderNo` |
| 支付成功 | `paySuccess` | 订单已完成，能量已委托 | `订单 {orderNo} 已完成\n能量：{energy}\n交易哈希：{txHash}` | `orderNo`、`energy`、`txHash` |
| 支付失败 | `payFailed` | 订单失败、超时或链上异常 | `订单 {orderNo} 失败\n原因：{reason}` | `orderNo`、`reason` |
| 钱包查询结果 | `walletQueryResult` | 用户点击「钱包查询」按钮后返回 | `地址：{address}\nTRX 余额：{balance}\n带宽：{bandwidth}` | `address`、`balance`、`bandwidth` |

> **「留空时使用默认文案」**：你不需要为每个场景都写文案——不填则 Bot 自动使用内置默认文案。编辑区下方会灰色小字提示当前场景的默认文案全文。

### 4.3 12 个变量占位符

下表权威源来自 `template-constants.ts` 的 `AVAILABLE_VARIABLES`，顺序与后端 `KnownVariables` 严格一致。

| 变量名 | 中文标签 | 含义 | 示例值 |
|---|---|---|---|
| `{orderNo}` | 订单号 | 系统生成的订单编号 | `ORD20260502123456` |
| `{packageName}` | 套餐名 | 当前套餐名称 | `5万能量包` |
| `{amount}` | 订单金额 | 订单应付金额（含单位） | `10.5 TRX` |
| `{energy}` | 能量数量 | 套餐对应的能量额度 | `65000` |
| `{address}` | TRON 地址 | 收能量地址或用户输入地址 | `TN7Yt...Rj9K` |
| `{payAddress}` | 付款地址 | 用户应付款到的 TRON 地址 | `TPayXxx...Yyy` |
| `{txHash}` | 交易哈希 | 链上委托交易的哈希 | `a1b2c3d4e5f6...` |
| `{botName}` | Bot 名 | Telegram Bot 的用户名 | `@energybot` |
| `{bandwidth}` | 带宽 | 账户当前带宽 | `5000` |
| `{balance}` | TRX 余额 | 账户 TRX 余额（含单位） | `12.34 TRX` |
| `{reason}` | 错误原因 | 失败、异常时的原因说明 | `余额不足` |
| `{command}` | 命令文本 | 用户发送的原始命令 | `/help` |

### 4.4 转义与占位规则

消息模板使用 `{name}` 语法的占位符，渲染规则权威源来自 `render.ts` 与后端 `template.go`：

| 输入写法 | 渲染结果 | 说明 |
|---|---|---|
| `{orderNo}` | 对应变量值 | 合法且已知变量，替换为实际值 |
| `{unknown}` | `{unknown}` | **合法但未知**的变量原样保留，便于你发现错别字 |
| `{{` | `{` | 转义：输出一个字面量左大括号 |
| `}}` | `}` | 转义：输出一个字面量右大括号 |
| `{a-b}` | `{a-b}` | 非法标识符（含 `-`、空格等），原样保留 |
| `{1x}` | `{1x}` | 非法标识符（数字开头），原样保留 |
| `{name` | `{name` | 未闭合占位符，原样保留 |
| 孤立的 `{` 或 `}` | 原样输出 | 不参与替换 |

> **不递归**：变量值中如果恰好含有 `{other}` 或 `{{escape}}`，不会被二次解析。

### 4.5 操作说明

1. **切换场景**：点击左侧 Tab，中部编辑区立刻切到对应场景。
2. **输入文案**：在 textarea 里编辑，支持多行（按 `Enter` 换行）。
3. **插入变量**：
   - 把光标定位在想插入的位置。
   - 点击中上方的变量徽章（推荐变量会高亮）。
   - 徽章对应的 `{name}` 会自动插入到光标处。
   - 也可以手敲 `{` 后输入变量名再 `}`。
4. **查看预览**：右侧预览区实时渲染：
   - **正常变量**：用示例值替换并高亮。
   - **未知变量**：红色背景标出。
   - **转义括号**：淡色标出，便于区分字面量括号和占位符。
5. **未知变量警告**：如果文案含未知变量，左侧场景 Tab 标题右上会出现红点，编辑区上方会弹出黄色警告条列出具体哪些变量未识别。
6. **保存**：消息模板**自动保存**——修改完后切 Tab 或离开页面会自动提交，右上角会弹出「操作成功」提示。

## 5. 常见问题（FAQ）

### Q1：保存菜单时提示「菜单嵌套深度不能超过 3 层」怎么办？

A：菜单最多只允许 3 层嵌套。检查你是不是在第 3 层里又放了 `submenu` 类型按钮——可以通过面包屑逐层检查每一层的按钮。把第 3 层的 `submenu` 按钮改成其他动作类型，或把业务拆到另一个入口。

### Q2：保存菜单时提示「套餐 ID 不属于当前 agent」怎么办？

A：通常是以下两种情况之一：

- 你把别的 agent 的配置复制过来，里面引用了不属于你的套餐。
- 你引用的套餐在「能量套餐管理」页被删除/转移了。

处理方式：编辑出错的「能量套餐组」按钮，从「套餐列表」里移除无效套餐，重新选当前 agent 下的有效套餐。

### Q3：保存提示「配置已被他人修改」怎么办？（409 冲突）

A：说明有其他同事在你打开页面后抢先保存了一次。处理方式：

1. **先备份你的改动**：打开一个记事本，把关键改动文字粘贴过去。
2. 点页面右上角「刷新」按钮（或 F5 刷新整页）。
3. 基于最新配置重新改一遍，再保存。

建议同一机器人同一时间只由一人编辑，避免踩这个坑。

### Q4：菜单已保存，但 Telegram 里的按钮还是旧的？

A：检查以下几点：

1. Bot 是否处于「运行中」状态？在顶部运行状态卡片确认。
2. 你发的消息触发的是否是同一个菜单层级？Telegram 里 `inline keyboard` 会跟随消息停留，发条新的 `/start` 拉起最新主菜单即可。
3. 极端情况下在 Telegram App 端下拉刷新对话或重开对话。

### Q5：Telegram 预览和实际发送的样式不完全一致？

A：预览是按 Telegram 的 `inline keyboard` 原生规则模拟的，颜色、圆角、字体在不同 Telegram 客户端（iOS / Android / Desktop / Web）上会有细微差异。预览主要用来验证**按钮布局和文本**，视觉细节以用户实际客户端为准。

### Q6：切换 Bot Token 后用户会感知到吗？

A：切 Token 相当于换一个 Bot 身份。你需要提醒用户到新 Bot 下重发 `/start`。后端会自动重启 Bot 进程，新 Token 立即生效。

### Q7：消息模板改了之后没找到「保存」按钮？

A：消息模板是**改完自动保存**，不需要手工点保存。保存成功时右上角会弹出「操作成功」toast。

### Q8：某个场景我不想自定义，留空会怎样？

A：留空时 Bot 会使用系统内置的默认文案（每个场景下方灰色小字展示的那一段）。你不必把 9 个场景都填满。

### Q9：能否看到文案里变量的实际值预览？

A：可以。右侧预览区会用示例值渲染（例如 `{orderNo}` 预览成 `ORD20260502123456`）。这些只是示例，不代表用户真实订单号。

### Q10：能否导出/导入整套配置？

A：当前版本**暂不支持**配置的导出导入。跨 agent 复制配置需要逐项手动复制。该能力在规划中。

## 6. 运维提示

### 6.1 切换 / 更新 Token 的影响

- 保存新 Token 后，Bot 进程会自动感知并重启（长轮询会重新建立连接，无需手动操作）。
- 老 Token 随后失效。如果你仍保留着老 Token 的引用（例如在外部 Webhook），需要同步更新。

### 6.2 v1 版本数据的兼容

本设计器是 v2 版本，与旧 v1 的 JSON 配置面板**不兼容**。升级部署时：

- `menu_config` / `message_config` / `welcome_text` 三个字段会被清空（部署 SQL 会处理）。
- 升级后需要**重新配置**一次菜单与消息模板。
- 「基础设置」Tab 的 Bot Token、用户名、备注等字段不受影响。

### 6.3 多人协作建议

- **同一 agent 由一人编辑**：避免乐观锁冲突。
- **先约定分工**：例如 A 负责菜单、B 负责消息模板，时间错开。
- **保存前沟通**：如果你即将做大改动，先在团队群里吱一声。

### 6.4 数据备份与恢复

后端数据库会定期备份，但**建议运营团队养成习惯**：

- 每次大改前，在记事本留一份关键文案的快照。
- 菜单设计涉及多层嵌套时，可以逐层截图保存为参考。

### 6.5 遇到疑难问题

1. 先按本手册 FAQ 自查。
2. 截图 + 描述复现步骤，反馈给技术对接人。
3. 紧急情况可先点「关闭机器人」止损，再慢慢排查。

## 附录：关键术语对照表

| 术语 | 含义 |
|---|---|
| Bot / 机器人 | 指一个 Telegram 机器人实例，由唯一的 Token 标识 |
| Agent / 代理商 | 本系统的使用方租户，一个 agent 名下可以有一个机器人 |
| 菜单 / Menu | 用户在 Telegram 里看到的按钮网格 |
| 按钮 / Button | 菜单的最小单元，对应一个 ButtonAction 动作 |
| 子菜单 / Submenu | 嵌套在按钮下的下一级菜单 |
| 模板 / Template | 带 `{变量}` 占位符的消息文本，运行时替换成实际值 |
| 占位符 / Placeholder | `{name}` 语法的变量插槽 |
| 乐观锁 | 提交时校验数据从你读取后是否被他人改过，有则拒绝（409） |
| dryRun | 仅校验不落库的预检模式 |
| TRX | 波场（TRON）链的原生代币 |
| 能量 / Energy | 波场链上的资源，用于抵扣合约执行费用 |
